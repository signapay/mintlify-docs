Every protected API endpoint requires an `Authorization` header. This tutorial walks through the two supported authentication schemes and the full session login flow, including multi-factor authentication.

## Authentication schemes

| Scheme | Header format | When to use |
| --- | --- | --- |
| Session token | `Authorization: Bearer <token>` | User-driven API calls. Obtained by completing the login flow below. |
| API key | `Authorization: Api-Key <api-key>` | Service-to-service integrations. Managed via the [Client API Key](/api-reference/clients/get-client-api-key) endpoints. |

For interactive integrations, use a session token. For automated back-end integrations, use an API key.

---

## Session login flow

### Step 1 — Create a session

Send a `POST /v1/crm/sessions` request with your email and password.

```http
POST /v1/crm/sessions
Content-Type: application/json

{
  "email": "you@example.com",
  "password": "your-password"
}
```

**Two possible responses:**

**A — MFA required.** The response contains a `challenge` object instead of a token. Proceed to [Step 2](#step-2-complete-mfa).

**B — Login successful.** The response includes a `token` field. Skip to [Step 3](#step-3-use-the-token).

---

### Step 2 — Complete MFA

If your account has MFA enabled, you will receive a one-time password via email or SMS. Submit it to `POST /v1/crm/sessions/mfa`.

```http
POST /v1/crm/sessions/mfa
Content-Type: application/json

{
  "challenge_id": "<challenge_id from step 1>",
  "code": "123456"
}
```

A successful response contains the session `token`. If the code expired or is incorrect, call [Resend MFA OTP](/api-reference/authentication/resend-mfa-otp) and try again.

---

### Step 3 — Use the token

Include the token in the `Authorization` header of every subsequent request.

```http
GET /v1/crm/sessions
Authorization: Bearer <token>
```

You can call [Get session](/api-reference/authentication/get-session) at any time to verify the token is valid and retrieve the associated user. Call [Get session permissions](/api-reference/authentication/get-session-permissions) to inspect the full permission set for the session.

---

## First login — forced password change

On first login or after an admin password reset, the API returns a `forced_password_change` flag instead of a usable token. You must set a new password before the session becomes active.

Call [Forced password change](/api-reference/authentication/forced-password-change) with the temporary credentials and your chosen new password, then complete the normal login flow.

---

## Password reset flow

If a user has forgotten their password:

1. **Initiate reset** — Call [Create password reset](/api-reference/authentication/create-password-reset) with the user's email. An OTP is sent to the registered address.
2. **Verify OTP** — Submit the code to [Verify password reset OTP](/api-reference/authentication/verify-password-reset-otp) to receive a reset token.
3. **Set new password** — Pass the reset token and chosen password to [Complete password reset](/api-reference/authentication/complete-password-reset).

Link-based reset flows (email link rather than OTP) use [Verify password reset token](/api-reference/authentication/verify-password-reset-token) in place of step 2.
